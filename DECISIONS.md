## 2026-06-20

### 投票顯示模式

決策：講師簡報只顯示大字即時結果與 QR Code；投票操作由學員手機頁完成。

理由：投影畫面需優先確保文字可讀與版面穩定，避免在同一區域同時放置操作按鈕、結果、QR Code 與長網址。

### Firebase 使用狀態

決策：使用 Firebase Anonymous Auth 與 Firestore 支援即時投票及目前開放題目同步。

理由：後續已加入學員手機投票需求；Firestore 僅保存 `sessions/default`、合法 poll id、匿名 vote 與必要時間戳，存取由 Security Rules 限制。

### 直接開檔模式

決策：簡報若以 `file://` 開啟，QR Code 固定指向 GitHub Pages 的公開學員投票頁，不使用本機檔案路徑。

理由：本機路徑無法供學員手機使用，且中文編碼後的長網址會造成投票版面溢出。

## 2026-06-21

### 選取流程與結論頁視覺結構

決策：第 5 頁以橫向四階段流程呈現選取思考；第 12 頁以「不是／而是」雙欄對比呈現結論。

理由：原垂直漏斗與大型段落造成版面失衡，無法清楚表達流程順序與概念轉折；新結構以位置、箭頭、色彩與對比強化認知關係。

### 投票數據一鍵歸零與重設計制

決策：採用「時間戳記邏輯重設 (Logical Reset)」機制，講師端 Hover 投票卡時可見「重新計票」按鈕，點擊後於雲端寫入 `resetTimestamp`。前後端只採計大於該時間戳的票數。

理由：避免講師在現場需要修改網址參數或跑 Node.js 腳本。此機制不需要刪除雲端歷史資料庫，既能安全地將前端顯示的票數一鍵清空並解鎖學員端 UI，又不會引發 Anonymous Auth 下惡意刪除他人資料的安全隱憂。
